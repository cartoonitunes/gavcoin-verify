### Constructor Behavior

The creation bytecode does the following:
1. PUSHes "GavCoin" (32 bytes, zero-padded): `7f476176436f696e00...`
2. PUSHes NameReg address: `73084f6a99003dae6d3906664fdbf43dd09930d0e3`
3. Calls `NameReg.register("GavCoin")` via selector `e1fa8e84`
4. Sets `owner = msg.sender` (from `owned`)
5. Sets `m_balances[owner] = tota` (from `coin` — likely 0)
6. Sets `m_lastNumberMined = block.number`
7. Copies runtime bytecode to memory and returns it

### Storage Layout

Based on inheritance order `BasicCoin, named("GavCoin"), coin("GAV", 1000), owned`:
- Slot 0: `m_balances` mapping (from BasicCoin)
- Slot 1: `m_approved` mapping (from BasicCoin)
- Slot 2: `owner` (from owned)
- Slot 3: `m_lastNumberMined` (from GavCoin)

Note: `sym`, `denom`, `tota` from the `coin` base contract may or may not occupy storage slots depending on how the preprocessor inlined them. If they're constructor-only values not stored, they won't appear in storage.

Verified on-chain: slot 3 contains the last mined block number (confirmed in the GavCoin dapp).

### The #require Preprocessor

The `#require` directive was part of the Mix IDE and Mist browser's Solidity tooling. It worked similarly to C's `#include` — it resolved contract names from a standard library and inlined them before compilation.

Sources:
- Standard library location: `ethereum/mix` repo, `stdc/` directory
- Contracts available: `owned`, `mortal`, `Config`, `NameReg`
- NOT found in standard library: `named`, `coin`, `service` — these may have been in a different location or generated by the preprocessor itself

### The `named` Contract

Reconstructed from bytecode:
- Constructor calls `NameReg(0x084f...).register(_name)` 
- `name()` returns the NameReg address (acts as a getter for the registry)
- `named(bytes32)` calls `NameReg.addressOf(bytes32)` to resolve names

### The `coin` Contract

Reconstructed from source analysis:
- Takes `(symbol, denomination)` parameters
- Sets variables: `sym`, `denom`, `tota`
- `tota` is referenced in GavCoin constructor as initial balance (likely 0)

### Key Challenges

1. **Preprocessor reconstruction**: The exact output of `#require` is unknown. The inlining order, whitespace, and structure may affect compilation.
2. **Compiler version**: 17 candidate versions (0.1.1 through 0.3.6). Each may produce different bytecode.
3. **Optimization**: Unknown whether optimization was enabled.
4. **Base contract structure**: `named` and `coin` are reconstructions, not originals.
5. **Solidity language changes**: Early versions had `hash()` instead of `sha3()`, `string32` vs `bytes32`, and other differences that changed between minor versions.

### Approach

1. Try compiling the flattened source with each candidate compiler version
2. Compare output bytecode with on-chain bytecode
3. Iterate on the flattened source structure based on compilation errors and partial matches
4. Use the creation bytecode for exact matching (includes constructor + runtime)
